WealthBridge Architectural Deep-Dive

A Step-by-Step Rationale of System Implementations, Technological Selections, and Alternatives Analysis

STEP 1

Raw Data Storage Intake (S3 Landing Bucket)

The entry point of the pipeline. Files containing transactions and positions are dropped here daily by upstream legacy platforms.

Why We Used It

Amazon S3 provides cloud-native, infinitely scalable object storage. It integrates out of the box with AWS lambda notification events and maintains a SLA durability of 99.999999999%.

Alternatives vs. Advantages

Alternative Pros & Cons Why S3 is Better for this Project
AWS EFS / Shared FS + Standard file access
- High cost, complex mounting 		S3 is significantly cheaper and emits native, direct events when objects are uploaded, which triggers our preprocessor.
Self-Hosted SFTP Server + Legacy compatibility
- Single point of failure, manual scaling 		S3 eliminates server management and scales automatically as file volumes grow or shrink.
STEP 2

Completeness Check Orchestration (AWS Lambda)

An event-driven Lambda function acting as a gatekeeper. It intercept S3 events and prevents triggering Glue until the file pair for a specific file_id is complete.

Why We Used It

Glue PySpark jobs take 1 to 2 minutes to spin up and cost money for every second of runtime. Utilizing a lightweight, fast Lambda function ensures we do not run Glue with missing files, avoiding wasted expenses.

Alternatives vs. Advantages

Alternative Pros & Cons Why Lambda is Better for this Project
Direct Glue Trigger + No Lambda needed
- Glue fails if second file is missing 		If only transaction is uploaded, Glue would boot up, search for position, fail, and bill you for the execution time. Lambda blocks this.
EventBridge Orchestrator + Pure serverless workflow
- Hard to do custom check logic 		EventBridge cannot query a database to check if a sibling file was uploaded hours ago. Lambda can.
STEP 3

State Tracking & Manifest Deduplication (PostgreSQL RDS)

Maintains pipeline status, execution audit logs, and file manifests. The Preprocessor Lambda checks this to detect duplicate uploads.

Why We Used It

Using relational tables with UNIQUE constraints ensures strict processing limits. It guarantees that the same file is never sent to Glue twice.

Alternatives vs. Advantages

Alternative Pros & Cons Why PostgreSQL is Better for this Project
Amazon DynamoDB + Fast, serverless
- Weak join capabilities, no relational rules 		We already store our data quality rules in the database. Placing the manifest there lets us run relational queries to check file pairs in one database call.
S3 Metadata Tagging + Zero database cost
- No cross-file indexing or query capability 		Cannot query S3 tags to ask: "Are there any unregistered files left in Batch 2?" PostgreSQL indexes this instantly.
STEP 4

ETL Execution Engine (AWS Glue PySpark)

The processing core. Ingests raw CSV data into Spark DataFrames, runs type casting, performs DQ validations, and writes outputs.

Why We Used It

AWS Glue offers a serverless Spark runtime. It provides scale-out parallel processing for massive files, native AWS Glue Catalog connections, and Apache Iceberg support.

Alternatives vs. Advantages

Alternative Pros & Cons Why AWS Glue is Better for this Project
Amazon EMR Cluster + Full cluster configuration
- High management overhead, pay for idle nodes 		EMR is expensive for episodic batch runs. Glue is serverless, boots up automatically, and scales to zero.
ECS Fargate (Python) + Very fast start times
- Single node limits, memory constraints 		Python containers crash if file sizes exceed RAM. Glue PySpark spills memory to disk and scales horizontally across multiple nodes.
STEP 5

Configuration-Driven DQ Engine (Spark SQL Expressions)

Validation engine reading rules dynamically from dq_rule_registry and evaluating them via Spark's native expr().

Why We Used It

Decouples business validation logic from Python code. Adding, updating, or deactivating a rule is a simple SQL row update rather than a code deployment.

Alternatives vs. Advantages

Alternative Pros & Cons Why our DB-driven Engine is Better
Code-Hardcoded Validations + Easy to code first time
- High maintenance; requires redeployment to fix 		With our engine, business users can update rules (e.g. changing the currency list) without developer resources.
Great Expectations Library + Feature rich, profiles data
- Complex configuration, slow runtime overhead 		Our engine uses native Spark SQL expressions, which compiled directly into optimized JVM bytecode, maintaining high speed.
STEP 6

Curated Data Table Format (Apache Iceberg)

The curated destination storage layer for validated clean and quarantined datasets.

Why We Used It

Apache Iceberg is a modern lakehouse table format. It provides ACID transactions, partition layout evolution (e.g. changing partition layouts without rewriting historical records), and time-travel history.

Alternatives vs. Advantages

Alternative Pros & Cons Why Apache Iceberg is Better
Raw CSV / Parquet + Simple format
- No transactions, slow updates, partition locks 		Parquet lacks ACID. If a write fails halfway, you get corrupted files. Iceberg guarantees atomic rollbacks.
Delta Lake + High speed, ACID
- Vendor lock-in (Databricks) 		Iceberg is open-source, fully supported natively by AWS Glue, Athena, Snowflake, and BigQuery without custom plugins.
STEP 7

Operational Correction Assistant (HTML Dashboard)

Automated output of dynamic reporting showing run outcomes, AI data narratives, and repair directives for anomalies.

Why We Used It

Saves hours of debugging. Operations teams get an immediate visual layout explaining exactly why records failed and what values to fix.

Alternatives vs. Advantages

Alternative Pros & Cons Why our HTML Report is Better
Database Logs Only + Zero code work
- Relies on SQL skills, slow lookup 		Business users and data operators cannot easily read raw database tables. The HTML report is graphical and self-explanatory.
Slack Alerts Only + Real-time notice
- Messages get buried, format is limited 		A Slack alert cannot easily hold a table of 50 bad records with separate repair steps. The dashboard organizes this in a clean format.
